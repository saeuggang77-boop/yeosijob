## Work Rules

- **변경 시 영향 범위 전체 검색 필수**: 코드 수정 전에 반드시 grep으로 관련 참조를 모두 찾고, 빠짐없이 한번에 수정할 것. 부분 수정 후 "끝났다"고 보고하지 말 것.
- **부작용 사전 고지 필수**: 기능 추가/변경 요청 시 그로 인해 발생할 수 있는 부작용(cascade 삭제, 데이터 정합성, 성능, 보안 등)을 미리 알려주고, 해결 방안을 함께 제시할 것. 사용자가 나중에 문제를 발견하게 두지 말 것.
- **더 나은 대안 먼저 제안**: 사용자가 방법을 제시해도, 더 좋은 방법이 있으면 구현 전에 먼저 제안할 것.
- **선제적 개선 제안 필수**: 기능 구현 시 "이게 실제로 자연스러운가?", "더 나은 방식이 있는가?"를 깊게 추론하고, 사용자가 묻기 전에 먼저 의견을 제시할 것.
- **실행 오류는 자동 해결**: bash 오류 등 기술적 오류는 사용자에게 묻지 말고 바로 수정해서 재시도할 것.
- **질문/의견에는 확인 먼저**: 사용자가 질문하거나 의견을 물으면 바로 실행하지 말고, 실행 여부를 먼저 확인할 것.

## QA Rules

- **"전체 QA 해"** = 코드 감사 + 로컬 런타임 QA + 프로덕션 런타임 QA 전부 실행
- **"코드 감사 해"** = 소스코드 정적 분석 (보안, 타입, 패턴)
- **"로컬 런타임 QA 해"** = 로컬 서버 띄우고 전체 페이지 curl 접속 테스트 (200/500 확인)
- **"프로덕션 런타임 QA 해"** = yeosijob.com 전체 페이지 curl 접속 테스트
- QA 시 반드시 `prisma migrate status`로 미적용 마이그레이션 체크 포함
- QA 시 반드시 `npm run build` 빌드 성공 여부 확인 포함
- 런타임 QA는 실제 curl로 모든 페이지 접속하여 500 에러를 잡아낼 것
